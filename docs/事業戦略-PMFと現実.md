# GUILD AI — 勝てる事業設計（理想ではなく現実）

> 制度設計者ではなく**ファウンダー視点**で書く。
> 数字感覚・PMF・収益視点・開発コスト視点で全振り。

---

## A. 厳しい批判 — GUILD AI 構想の致命的弱点

### A-1. **MD ファイルは商品として弱い**

買い手が払うのは「文章」ではなく「**動く成果**」。
- 個人開発者は GitHub Copilot で月 $20 払って動くコードを得る
- 法人は ChatGPT Enterprise で月 $30/人 払って即時生産性向上
- どちらも「文章を読んで自分で実装する」のは**負けの構造**

→ 現状の MD ファイル販売は「**読んで応用してね**」という buyer に労働を渡す商品。**動かない MD は売れない**。

### A-2. **OpenAI / Claude / Gemini は買わない**

これは断言する。理由：
- 彼らは**自前で合成データを作る方が安い**（年予算 ¥1000 億超のチームが内製）
- 第三者から買うと**ライセンス・著作権・データ provenance のリスク**
- 単価モデル：API 1 call あたり ¥0.01 のロイヤリティでも、月 100 億 call で ¥10 億の追加コスト → 内製と比較にならない
- 彼らが欲しいのは **fine-tuning 用の数百万件規模のデータセット**であって、1000 件の MD 集ではない
- 既存の data partnership は **Reuters / FT / Stack Overflow / Reddit** と直接契約。中間マーケットを通さない

→ **OpenAI 等を期待するな**。期待すると戦略がブレる。

### A-3. **個人作家市場は小さい・遅い**

note / Zenn / Qiita のクリエイター経済を見れば分かる：
- 月 ¥10 万稼ぐクリエイターは全体の 1% 未満
- 平均は月 ¥500-3000 程度
- 著者の参入インセンティブ ≈ 趣味 + 副業お小遣い
- 結果、**プロフェッショナル品質のサプライが集まらない**

→ 個人著者依存では市場形成に 5-10 年かかる。**法人発で需要を作ってから個人に逆流させる**順序が現実的。

### A-4. **評価制度の過剰設計**

v1〜v4 で書いた制度は **0→1 では絶対に不要**。
- バズる前に「不正耐性」を考えても、不正する人がそもそもいない
- prediction market は流動性が無いと機能しない（参加者 1000 人未満）
- Capability Uplift benchmark は計測コストが製品単価を超える可能性

→ **MVP では評価制度は手動キュレーションで十分**。100 packs 以下なら founder が全部読んで決めれば良い。

### A-5. **「AI が買う市場」は 5 年早い**

A2A（AI-to-AI）取引はまだ研究領域。
- 2026 年時点で AI agent が自律的に課金できる仕組みは普及していない
- 結局、買い物決定するのは**人間の中間管理職**
- 「AI が買う」前提の制度設計は今は早い

→ **人間の購買担当者**を最初の UI で前提にする。AI 自動購入は後付けで足す。

### A-6. **DB 移行・auth 等の技術投資は GUILD AI の差別化ではない**

ここまで作った DB 移行・auth・session 管理は**事業差別化要因ではない**。
どこでも作れる。**事業価値は集まったコンテンツとそれを売る顧客接点**にある。

→ 技術スタックに時間使いすぎ。**サプライ集めと顧客接点に時間を振り向けるべき**。

### A-7. **市場規模の検証が無い**

ここまで「世界を取る」と言ってきたが、**TAM/SAM/SOM の数字がない**。
- TAM（理論最大）: 世界の AI 開発投資 ≈ 数十兆円。だが GUILD AI が取れる% は？
- SAM（実現可能）: 日本の AI 受託会社市場 ≈ 数千億円
- SOM（初期狙える）: 日本の AI 受託会社で「外部知識を月 ¥10 万で買いたい」会社 ≈ 1000 社程度
  - そのうち実際に契約する確率 5% = **50 社**
  - 50 社 × ¥10 万/月 = **月 ¥500 万** → 年 ¥6000 万

→ **個人市場と法人市場で TAM が 100 倍違う**。法人を最初に狙うのが合理的。

## B. 本当に売れる MVP — 月商 100 万円までの最短ルート

### B-1. 商品の再定義：MD ではなく「**Skill Pack**」

| 旧（MD） | 新（Skill Pack） |
|---|---|
| 知識ファイル 1 枚 | **動くもの一式**: prompt + コード + テスト + 説明 + benchmark 結果 |
| ¥1,000-5,000 | ¥30,000-300,000 |
| 「自分で読んで実装してね」 | 「**コピペすれば動く**」 |
| 著者 = ブロガー | 著者 = 実務エンジニア・コンサル |
| 評価 = 文章採点 | 評価 = **AI 性能差分**（before/after） |

具体例（受託会社が欲しがる Skill Pack）：

1. **「Stripe Subscription + Next.js + 日本語請求書」 Skill Pack** — ¥80k
   - 動く Next.js コード一式
   - 日本特有の請求書要件対応プロンプト
   - 法人決済テストケース
   - 想定買い手：SaaS 立ち上げ受託する制作会社

2. **「医療系問い合わせ Bot 構築」 Skill Pack** — ¥200k
   - 医療特有用語辞書
   - 薬機法回避プロンプト
   - エスカレーション分岐ロジック
   - 想定買い手：医療業界向け AI 受託会社

3. **「不動産査定 AI」 Skill Pack** — ¥150k
   - 不動産用語データセット
   - 査定ロジックプロンプト
   - 法令ガード（宅建業法）
   - 想定買い手：不動産 SaaS 立ち上げ会社

これなら「文章読む」のではなく「**コピペで動く**」。買い手の労働ゼロ。

### B-2. 削るべき機能（**全部削る**）

| 機能 | 削る理由 |
|---|---|
| Prediction market / TCR / Challenge | 流動性なし、参加者ゼロでは機能しない |
| Knowledge Equity Tokens | 法的グレー（金商法）。流通市場が無い |
| Capability Uplift Benchmark | 計測コストが商品単価を超える |
| Citation PageRank | 100 packs 以下では trivial |
| 多言語対応 | 日本語 LP1 本でまず売れるか確認 |
| author_reputation の動的更新 | 100 著者以下なら手動で十分 |
| A/B テスト基盤 | 検体が少なくて統計的に意味なし |
| C ランクの自動分類 | 100 packs 以下なら founder が手動で除外 |
| Negative Selection | 投稿数 1000 以下では人力レビュー |
| dynamic policy / market_phase | パラメータいじる前に PMF 検証 |
| middleware redirect | 後でいい（既に実装済みだが拡張不要） |
| FK constraint enforcement | 既存テストを壊すだけ。後でいい |

→ **コードの 70% は当面捨てる**。MVP の必要機能は：
- 商品リスト（Skill Pack 一覧）
- 購入（Stripe 決済）
- ダウンロード（zip 配布）
- 買い手レビュー（5 ☆ + コメント）
- 著者ダッシュボード（売上 + ダウンロード数）
- 認証（既に実装済）

これだけ。**1-2 ヶ月で実装可能**。

### B-3. 月商 100 万までの数字

```
顧客 30 社 × 月額 ¥30,000 (Pro Buyer 月額制)
= ¥900,000 MRR

OR

5 件の Skill Pack × ¥200,000 一回購入 / 月
= ¥1,000,000

OR

法人契約 2 社 × ¥500,000/月（unlimited license）
= ¥1,000,000 MRR
```

**最短は法人契約モデル**。2 社契約取るだけ。

## C. 買い手の絞り込み — 最初の顧客は誰か

### 結論：**AI 受託会社・SI**（最初の 100 顧客の 8 割）

なぜか：
1. **彼らは時間で稼ぐ**：月単価 ¥80-150 万で人を貸す。1 日でも工数削減できれば ROI 即時
2. **彼らはドメイン知識を持たない**：医療案件取ったが医療を知らない。テンプレート買って即対応したい
3. **彼らには予算がある**：プロジェクト原価の 5-10% は外注ツール代に充てる
4. **意思決定者が技術部長 / プロジェクトマネージャ**：技術を見て即決できる
5. **顧客対象として認識可能**：会社名で訪問可能。名簿あり

### 二次顧客：中小 SaaS / AI スタートアップ
- 自社プロダクトに AI 機能組み込みたい
- 内製エンジニアがドメイン専門知識を持たない
- 月額 ¥30k-100k 程度なら払う

### **個人開発者は最初は無視**
- 単価が低すぎる（¥3k/pack）
- 100 件売って ¥30 万 → 法人 1 社契約と同じ
- 個人を狙うとサポートコストで赤字

### 何故金を払うか — 言語化

```
受託会社の社長 / PM が抱える課題:
「医療案件取ったけど、社内に医療知識者ゼロ。3 ヶ月の調査に 2 人月 ≈ ¥200 万かかる」

GUILD AI Skill Pack の提案:
「医療 AI Skill Pack ¥200k 買えば、3 ヶ月分の調査が 1 週間に短縮できる」

ROI: ¥200k 投資で ¥1500k 削減 → 7.5 倍リターン
```

これが**営業トーク**。受託会社は数字で動く。

## E. 評価制度の現実版（極限までシンプル）

### 初期 100 packs まで：**手動キュレーション**

founder が全 pack を読み、以下の質問だけで S/A/B 判定：

1. **コピペで動くか**（自分で試す）
2. **ベンチマーク結果が嘘でないか**（再現する）
3. **業界知識が正しいか**（専門家にレビュー依頼 ¥10k/件）

それ以上の制度は不要。S=Founder 直推 / A=売れている / B=出てるだけ。

### 100-1000 packs：**売上ベースの自動分類**

```
S: 直近 90 日で 10 件以上売れた + 平均 ☆ 4.5 以上 + 返金率 < 5%
A: 直近 90 日で 3 件以上売れた + 平均 ☆ 4.0 以上
B: それ以外
```

**たったこれだけ**。複雑な軸は不要。市場が答えを出す。

### 1000+ packs：v3 設計を導入

ここで初めて downstream_success などの指標が意味を持つ。それまでは作らない。

## F. 収益モデルの現実

### 最短：**Concierge 型販売**（営業ドリブン）

founder が直接受託会社に営業：
- 「あなたの会社の困っているドメインの Skill Pack を作ります」
- カスタム制作料 ¥500k-2M / 件
- マージン 60%
- → **5 件で月商 ¥1500-1500 万**

これが **0→1 で月商 100 万を超える最速ルート**。マーケットプレイスじゃなくて受託。

### スケール：**法人サブスク**

Skill Pack 在庫が増えてきたら：
- 月額 ¥100k で全 Skill Pack アクセス可能
- 100 社 = 月商 ¥1000 万
- 営業コスト：1 営業 = 月 30 件契約取れる前提

### スケール 2：**マーケットプレイス手数料**

著者が増えたら：
- 取引額の 20% を運営取り
- 月流通額 ¥5000 万 → 運営収益 ¥1000 万

### 最後に：**ホワイトラベル**

大手企業（楽天 / リクルート / 富士通 等）に：
- "御社版の AI スキル市場を作りませんか" と提案
- 開発・運営委託で年 ¥3000 万-1 億
- → 大型契約

### **広告・コンサル等は最初は売らない**

集中すべきは Skill Pack 販売。それ以外は気が散る。

## G. OpenAI / Claude / Gemini が買う未来

### 結論：**買わない。期待するな**

理由は A-2 で述べた通り。

ただし**例外はある**：
- **Anthropic / Google 等の data partnership 担当**が個別契約として一括買い取り（年 ¥1 億等）
- これは GUILD AI が**特定ドメインで世界一のデータベース**になった場合のみ
- 例えば「日本の医療法規データセット」で世界最大 → Anthropic が "Claude for Japan healthcare" のために買う
- 確率は 5 年以内に 10% 程度

→ **戦略の中心に据えるな。ボーナスとして起こり得るとだけ認識**

代替戦略：**OSS Agent 経由で間接的に流通**
- LangChain / AutoGPT / OpenAgents 等の OSS プラットフォームに **GUILD AI 統合プラグイン**を提供
- agent が必要な Skill Pack を自動購入する仕組み（agent operator が支払い）
- これは現実的に 2027-2028 で可能

## I. 最終結論 — 断言

### GUILD AI は勝てるか

**勝てる。ただし条件付き。**

### 勝てる形

```
1. MD 販売ではなく「Skill Pack マーケットプレイス」
2. 個人ではなく「AI 受託会社・中小 SaaS」を顧客にする
3. 評価制度ではなく「Concierge 型販売」で 0→1
4. グローバルではなく「日本の B2B 受託会社」で 1 億達成
5. AI 自動購入ではなく「人間の購買担当者」を UI 設計の中心
6. 高度な制度設計ではなく「営業 + 5 ☆ レビュー + 売上ランキング」
```

### 今すぐ捨てるべきもの

| 捨てるもの | 理由 |
|---|---|
| Prediction market / TCR | 100 packs 以下で機能しない |
| Knowledge Equity Tokens | 法的リスク + 流通市場なし |
| Capability Uplift Benchmark | 計測コスト > 商品単価 |
| 多言語対応 | まず日本語で売れるか検証 |
| 個人開発者市場 | 単価低すぎ |
| OpenAI 等への営業 | 買わない |
| 複雑な評価制度（v3, v4） | サプライ100以下では無意味 |
| middleware / DB 設計の追加投資 | 差別化にならない |

### 今すぐ集中すべきもの

| 集中するもの | 理由 |
|---|---|
| **30 社の受託会社にコールド営業** | 顧客の声 = すべて |
| **5 つの Skill Pack を founder 自ら制作** | 商品が無いと何も始まらない |
| **Stripe 決済 + ダウンロード機能のみ実装** | 売る箱があれば良い |
| **5 ☆ レビュー + 売上ランキング** | 評価はこれだけで足りる |
| **Concierge 受注（¥200-500k/件）** | 0→1 の最速ルート |

### 一行で言うと

> **「制度を作るな、客に会え。」**

理論を磨くより、**3 月までに最初の有償顧客を 5 社獲得する**ことに全リソースを振り向けろ。
それ以外は全部後回し。

## 補足：今ローカルで動いている技術スタックの活用

ここまで作った技術は無駄ではない：
- ✅ **Auth + 個人情報登録** → 受託会社の購買担当者向け B2B account として使える
- ✅ **DB 移行（Drizzle/Neon）** → スケール時に活きる、今は overkill
- ✅ **Server Actions + middleware** → 法人ダッシュボード作る時に活用
- ⚠️ **MD 評価エンジン v1-v4** → MVP では使わない（手動でいい）
- ⚠️ **escrow / api-gateway / atoa-*** → A2A 機能、5 年早い

技術投資は「**持っておくが今使わない**」状態でストックして、**売れる仕組み作りに振り向ける**。
